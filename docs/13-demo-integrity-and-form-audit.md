# Demo Integrity and Form Audit

**Reviewed:** 19 July 2026  
**Scope:** Version-one session-backed demonstration application

## Outcome

The demo now treats identifiers according to who controls them. Portal-owned identifiers are generated by the API. External identifiers are captured from their source document or equipment plate. Classification values are selected from controlled lists.

## Identifier ownership

| Record or field | Behaviour | Reason |
|---|---|---|
| Item SKU | Generated as `DEMO-ITM-####` | Portal-owned catalogue identity |
| Site code | Generated as `DEMO-STE-####` | Portal-owned site identity |
| Location code | Generated as `DEMO-LOC-####` | Portal-owned location identity |
| Asset number | Generated as `DEMO-AST-####` | Portal-owned asset identity |
| Transfer reference | Generated as `DEMO-TRF-####` | Portal-owned workflow reference |
| Count reference | Generated as `DEMO-CNT-####` | Portal-owned workflow reference |
| Movement reference | Generated by movement type | Portal-owned immutable ledger reference |
| Operating business | Selected from Phatsema Projects & Supplies or Phatsema Mining | Prevents free-text internal codes; legal relationship still requires client confirmation |
| Country and timezone | Selected from controlled lists | Prevents spelling and formatting errors |
| Manufacturer serial number | Entered by user | External equipment identity |
| Vehicle registration | Entered by user | External regulatory identity |
| Supplier or delivery reference | Entered by user | External source-document identity |
| Project, job, or cost-centre reference | Entered by user | External operational identity |
| Batch code | Entered when batch custody is enabled | Supplier or manufacturer identity |

Generated identifiers are created on the server, not in the browser. This avoids duplicate values when multiple users create records at the same time and keeps the API authoritative.

## Form audit

The following create and workflow forms were reviewed:

- login;
- user creation, role assignment, site scope, activation, and temporary password reset;
- site and location creation;
- catalogue item creation and editing;
- inventory receipt, issue, adjustment, and reversal;
- transfer creation, approval, dispatch, receipt, and cancellation;
- stock-count creation, entry, review, recount, approval, and posting;
- asset creation, assignment, status, meter, and service information;
- report filters and audit search.

The forms now use generated internal identifiers, controlled choices for classifications, required-field validation, field length limits, decimal-safe quantities, and API error summaries. Optional references remain optional only where the workflow can be trusted without them.

## Inventory integrity controls

- Site-scoped users can only read and mutate their assigned sites.
- Available stock is calculated from on-hand less reserved and quarantined quantities.
- Transfer approval reserves stock.
- Dispatch releases the reservation, removes source stock, records destination in-transit stock, and writes a dispatch ledger movement.
- Receipt reconciles received and rejected quantities, requires discrepancy reasons where applicable, clears in-transit stock, adds received stock, and writes a receipt ledger movement.
- Transfer movements cannot be reversed through the generic adjustment flow.
- Stock cannot move through a location while an active count freezes it.
- Count approval follows maker-checker separation and requires a review note for variances.
- Transfer requesters cannot approve their own transfers.
- Quantities retain three decimal places without float rounding.
- Duplicate item names, site names, location names within a site, user emails, and manufacturer serial numbers are rejected.
- Updates use record versions to reject stale writes.
- Mutations create audit events.

## User and access controls

The demo supports system administrator, operations manager, site manager, storekeeper, and executive viewer roles. Permissions are deny-by-default, and routes enforce both capability and site scope. Administrators can create users, assign a role and site scope, activate or deactivate accounts, and issue a temporary password. A user cannot deactivate their own account.

## Deliberately deferred

The following capabilities should not be invented from website marketing copy. They require confirmation in the requirements workshop:

- purchase requisitions, purchase orders, suppliers, goods-received matching, and invoice matching;
- formal legal-entity, VAT, and combined-billing rules for Agape Water and Phatsema;
- full batch, lot, expiry, and serial custody;
- client-owned stock custody and consignment accounting rules;
- rental contracts, hire billing, off-hire, and return inspection;
- preventive maintenance work orders and spare-part consumption;
- barcode labels, scanning hardware, offline operation, and conflict resolution;
- approval thresholds and delegations;
- MariaDB production persistence, backups, recovery, and data migration;
- finance, payroll, telematics, and document-management integrations.

These are recorded as discovery items, not silently treated as completed requirements.

## Quality evidence

The implementation is covered by frontend unit tests, backend feature tests, OpenAPI validation and generated types, fixture validation, strict TypeScript, ESLint, PHP parsing, Laravel Pint, PHPStan, and a production Vite build. Browser acceptance tests remain part of the release gate whenever the frontend and PHP server are running together.
