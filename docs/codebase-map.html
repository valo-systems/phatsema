1. Three parts, one repository

The whole system is one repository with three top-level pieces. They ship together and deploy to a single web address.

apps/web

The screens

  • React applicationEverything the user clicks
  • Builds to static filesNo server needed to run it
  • Talks only via the APINever touches data directly

packages/contracts

The agreement

  • openapi.yamlDefines all 71 endpoints
  • Generated typesFrontend types come from here
  • Written firstBefore either side is built

apps/api

The rules

  • Laravel applicationDecides what is allowed
  • Owns every business ruleThe screens cannot bypass it
  • Holds the demo dataIn session, no database

2. The contract is the spine

This is the single most important relationship. The contract is written first; the frontend's types are generated from it, so the two sides cannot drift apart without the build failing.

openapi.yamlHand-written. The source of truth.
→
generated/api.d.tsGenerated. Never edited by hand.
→
Every frontend API callTyped against the contract.
Why this matters in one sentence: if someone changes the API without updating the contract, the frontend stops compiling — so mismatches are caught at build time instead of in front of a client.

4. Follow one real request

Pick a flow and follow it from the click to the saved record. These are the actual files involved.

The user fills in the form

apps/web/src/features/inventory/ReceivePage.tsx

Site, location, supplier reference, and item lines. Controls come from the shared folder, so this page defines no inputs of its own.

They confirm on a review step

Posting stock cannot be undone, so the form shows exactly what will change before anything is sent.

The feature sends the request

apps/web/src/features/inventory/api.ts

Each feature owns its own data calls. It uses the shared client at shared/api/client.ts, which attaches the security token.

The contract types the call

packages/contracts/openapi.yaml

The shape of this request was agreed here first. A wrong field name fails to compile.

The route receives it and checks permission

apps/api/routes/web.php

POST /api/v1/receipts requires the inventory.receive permission before the request goes any further.

Input is validated

apps/api/app/Http/Requests/Inventory/ReceiptRequest.php

Runs before any business code. Bad input never reaches the rules.

The controller hands off immediately

apps/api/app/Http/Controllers/Api/V1/StockActionsController.php

It contains no business logic — it only passes the request on and returns the result.

The use case does the work

apps/api/app/Application/Inventory/StockCommandService.php

Checks the user may act on that site, generates a reference, applies the stock rules, writes ledger entries, and records an audit event — all inside one transaction.

Storage is written through an interface

apps/api/app/Domain/Inventory/InventoryRepository.php

The use case asks for an interface, never a specific database. This is what makes the storage swappable.

The demo store fulfils it

apps/api/app/Infrastructure/DemoStore/DemoStore.php

Saves into the browser session. Replace this one class with a database version and nothing above changes.

The screen refreshes itself

The response returns as JSON, cached lists are marked stale, and balances and the ledger update automatically.

The URL holds the view

apps/web/src/shared/hooks/useTableState.ts

Search, filters, sorting, and page number all live in the web address — so any view can be copied and shared, and the back button works.

One shared toolbar

apps/web/src/shared/ui/data/DataToolbar.tsx

Every list page uses the same search and filter bar. On a phone the filters collapse into a drawer.

One shared table

apps/web/src/shared/ui/DataTable.tsx

Sorting, paging, loading skeletons, empty and error states, and CSV export all live here once, not on each page.

The backend filters by permission

The list is trimmed to the sites this user may see. Hiding a button on screen is convenience; this is the real enforcement.

The point: ten list pages, one table and one toolbar. A fix to either improves every page at once.

Sign-in screen

apps/web/src/features/auth/LoginPage.tsx

In demo mode it also offers the five personas, so a reviewer can switch roles without needing accounts.

The server creates a session

A secure cookie is set. No token is stored in the browser where a script could read it.

The app loads who you are

apps/web/src/shared/auth/session.ts

Name, roles, permissions, and assigned sites. The navigation is built from this, so people only see what they can use.

Every page is guarded

apps/web/src/app/guards.tsx

Signed-out visitors are sent to sign-in and returned to where they were headed. If the session expires mid-use, the app says so rather than failing silently.

The backend checks again anyway

apps/api/app/Policies/

Six policies, deny by default. Every single request is re-checked; the screen is never trusted.

5. The one rule that keeps it tidy

Dependencies point in one direction only. This is what stops the codebase turning into a knot.

This part…may use…may not use…
A feature
features/inventory		Shared — controls, table, client, formattingAnother feature's internals. Inventory must not reach inside transfers.
Shared
shared/		Other shared code onlyAny feature. If shared needed a feature, it would not be shared.
ControllerApplication servicesBusiness rules. Controllers translate; they do not decide.
ApplicationDomain interfacesA specific database. It asks for storage, never names one.
DomainNothing outside itselfLaravel, HTTP, storage. Pure rules, so they are easy to test.
How this is enforced, not just intended: raw dropdowns, text boxes, checkboxes and buttons are banned everywhere except shared/ui/controls/. The rule runs in the build, so the design system cannot quietly drift.

6. The swap point

The demo has no database. Understanding this one line explains how it becomes a real system.

apps/api/app/Providers/AppServiceProvider.php

  InventoryRepository::class   ┐
  CatalogueRepository::class   │
  TransferRepository::class    ├──→  DemoStore   # today: browser session
  AuditRepository::class       │
  …14 interfaces in total      ┘

Fourteen storage interfaces currently all point at one class. Point them at a database version instead and the swap is complete — no controller, rule, screen, or contract changes. That portability was the main architectural requirement, and it is why the demo is not throwaway work.

Worth saying out loud in a demo: data lives only in the browser session. Closing the browser or pressing “Reset demo” restores the original example data. That is deliberate, so anyone can experiment without consequences — and so no one mistakes example figures for real stock.
